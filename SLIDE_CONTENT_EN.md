## Slide 03 — What is Generative UI?
**Template:** G (Big statement)

**Big quote:**
```
"UI composed by AI
at runtime — not hardcoded."
```

**Comparison table below:**

| | Traditional UI | Generative UI |
|---|---|---|
| Design | Defined upfront by designers | Generated by AI at runtime |
| Structure | Fixed, same for everyone | Flexible, personalized |
| Changes | Requires a new deploy | Adapts instantly to context |
| Experience | One-size-fits-all | Optimized per individual |

## Slide 05 — Demo 1 — How It Works
**Template:** F (Flow)

**Heading:** `HOW IT WORKS — STRUCTURED OUTPUT`

**Flow diagram:**
```
User input: "book a plumber"
        │
        ▼
POST /api/genui/form
        │
minimax/minimax-m2.7  (via OpenRouter)
        │
  Text stream — raw JSON embedded in markdown
        │
        ▼
extractJson()   strips markdown, parses {...}
        │
        ▼
Zod FormSchema.safeParse()
        │
        ▼
FormRenderer   maps fields[] → HTML inputs
```

**Key callout:**
```
No tool calling — AI returns JSON as plain text.
The prompt defines the schema shape.
More portable across models, less reliable than tool calling.
```

## Slide 07 — Demo 1 — 3-Mode Comparison
**Template:** H (full-width table)

**Heading:** `3 MODES · 3 DIFFERENT COST PROFILES`

| | MODE 1 — FE STATIC | MODE 2 — BE SDUI | MODE 3 — GENUI |
|---|---|---|---|
| **Latency** | ~5ms | ~50ms | ~3,000ms |
| **Cost** | $0 | $0 | ~$0.001/req |
| **Determinism** | 100% | 100% | ~80% |
| **Who controls** | Developer | Admin/Business | AI |
| **Failure mode** | None | None | Zod validation |
| **Use when** | Schema known upfront | Admin needs to edit without deploy | Schema can't be pre-authored |

**Highlight MODE 3 row** with purple border.

```
Confusing these 3 modes is why teams spend
GenUI budget on problems that SDUI solves better.
```

## Slide 09 — Demo 2 — How It Works
**Template:** F (Flow)

**Heading:** `HOW IT WORKS — VERCEL AI SDK v6`

**Flow diagram:**
```
useChat() + DefaultChatTransport
        │
        ▼
POST /api/chat
        │
streamText({ model, tools: 5 Zod schemas })
        │
toUIMessageStreamResponse()
        │
        ▼
m.parts[]
  ├── type: "text"        → prose stream (word-by-word)
  └── type: "tool-input"  → args buffer
                                │
                         args complete
                                │
                         renderTool(type, input) switch
                                │
                         Component renders inline
                         (no execute() — client-side only)
```

**Key callout:**
```
Vercel AI SDK handles the entire streaming protocol.
Tools are Zod schemas — no server-side execution needed.
Add a new component type = add 1 tool definition.
```

## Slide 14 — Demo 3 — How It Works
**Template:** F (Flow)

**Heading:** `HOW IT WORKS — RAW SSE STREAMING`

**Flow diagram:**
```
buildSignalBundle(persona, scenario)
  → persona · time · weather · location · behavior
        │
        ▼
POST /api/compose
        │
anthropic/claude-3-5-haiku   tool_choice: "auto"
        │
  SSE events (raw fetch, no SDK):
  ├── { type: "text" }   → reasoning panel (green)
  ├── { type: "tool" }   → Zod safeParse → layout update
  └── { type: "done" }   → metrics display
        │
4 tools: restructureHome · showMoodPicker
         showContextBanner · showCuratedRow
        │
        ▼
safeParse success → typed layout args applied
safeParse fail    → raw JSON fallback (never drops)
```

**Key callout:**
```
No SDK — raw fetch + manual SSE parsing.
Claude 3.5 Haiku: only model that streams text + tools in the same response.
safeParse fallback = UI is never blank even when validation fails.
```

## Slide 15 — Demo 3 — Signal Bundle & Why Not a Rule Engine
**Template:** E (2 column)

**Heading:** `THE PROBLEM: LAYOUT THAT ADAPTS TO CONTEXT`

**Header (full width — above 2 columns):**
```
INPUT — Signal Bundle
persona · time · weather · location · recentOrders · searchHistory
→ Data every consumer app already has. No new infrastructure needed.
```

**Left column — Rule Engine (the obvious first approach):**
```
RULE ENGINE
Write if/else for every signal combination:

if weather < 18°C AND persona == "budget"
  → show comfort_cheap_row

if weather < 18°C AND persona == "family"
  AND time > 18:00
  → show family_warm_combo

...

4 personas × 3 scenarios
× weather × behavior × time
= hundreds of rules
= impossible to maintain
```

**Right column — GenUI:**
```
GENUI
Replace all rules with 1 prompt:

"Persona Minh, 28, HCM,
weather 16°C cold, 19:30,
searchHistory: [pho, beef soup]..."

→ AI decides the tool calls

Add a new signal?
→ Add 1 line to the prompt.
   No new rules to write.
```

**Callout below:**
```
Trade-off: non-deterministic output.
Mitigation: cache layout per session, re-trigger only when signals shift significantly.
```

## Slide 17 — When to Use What?
**Template:** H (full-width table)

**Heading:** `ESCALATION PATH — NOT EVERYTHING NEEDS GENUI`

| Situation | Use |
|---|---|
| Fixed form, schema known upfront | **FE Static** |
| Admin needs to edit schema without a deploy | **BE SDUI** |
| Regulated domain (KYC, insurance, medical) | **BE SDUI** |
| Free-text → form, long-tail trade schemas | **GenUI — Demo 1 pattern** |
| Chat with many response types (chart/card/form) | **GenUI — Demo 2 pattern** |
| Consumer app, many signals, layout needs restructuring | **GenUI — Demo 3 pattern** |
| N × M combinations of personas × signals | **GenUI — Demo 3 pattern** |

**Callout below:**
```
Rollout order:  Static → SDUI → GenUI
                Only step up when you've hit the wall below.
```
